# 알람 동작 스펙

## 데이터 모델

```ts
type Alarm = {
  id: string        // 고유 ID
  name: string      // 알람 제목 (최대 50자)
  interval: number  // 주기 (일 단위, 최소 1)
  createdAt: string  // 시작일 — 갱신 시 현재 날짜로 리셋됨
}
```

---

## 알람 생명주기

```
[생성] → [진행 중] → [만기] → [갱신] → [진행 중] → ...
```

### 1. 생성

- 사용자가 제목과 주기(일)를 입력하면 알람이 생성된다.
- 시작일은 생성 시점의 날짜가 된다.
- 생성 직후 알림이 스케줄링된다.

> 예: "칫솔 교체"를 주기 90일로 등록하면, 90일 후가 만기일이 된다.

### 2. 진행 중

경과일이 주기보다 작은 상태.

- 프로그레스바 = 경과일 / 주기
- 남은 일수 = 주기 - 경과일

> 예: 주기 30일, 시작일로부터 15일 경과 → 프로그레스바 50%, 남은 일수 15일

### 3. 만기

경과일이 주기 이상인 상태.

- 프로그레스바 = 꽉 참 (100%)
- 남은 일수 = 0일
- **갱신 버튼을 누를 때까지 이 상태가 유지된다.** 자동으로 다음 사이클로 넘어가지 않는다.

> 예: 주기 30일, 시작일 2/20, 오늘 4/20 (59일 경과) → 3/22에 만기였지만 갱신을 안 했으므로 여전히 프로그레스바 100%, 남은 0일

### 4. 갱신

- 갱신 버튼을 누르면 시작일이 **오늘**로 리셋된다.
- 프로그레스바가 0으로 돌아가고, 남은 일수가 주기만큼 다시 설정된다.
- 알림도 새 만기일 기준으로 재스케줄링된다.

> 예: 위 상태에서 4/20에 갱신 → 시작일 4/20, 프로그레스바 0%, 남은 30일, 다음 만기일 5/20

---

## 경과일 계산

```
경과일 = floor((오늘 자정 - 시작일 자정) / 1일)
```

- 시간은 무시하고 날짜 단위(자정 기준)로만 계산한다.
- 시작일은 오늘 이전 날짜만 선택할 수 있다.

## 알림

### 스케줄링 규칙

- 만기일과 그 다음 1일에 알림을 예약한다.
- 각 날짜마다 **오전 8시, 오후 6시** 2회 발송한다.
- 이미 지난 시간은 건너뛴다.
- 만기 상태에서 갱신을 안 한 경우, 오늘 기준으로 알림을 예약한다.

> 예: 주기 30일, 시작일 4/1 → 만기일 5/1. 5/1 오전 8시, 오후 6시 + 5/2 오전 8시, 오후 6시에 알림 예약

### 재스케줄링

다음 동작이 발생하면 기존 알림을 전부 취소하고 전체 재계산 후 재예약한다:

- 알람 생성 / 수정 / 삭제 / 갱신
- 알림 토글 on/off

### 알림 권한

- 첫 알람 생성 시 OS 알림 권한을 요청한다.
- 권한이 거부되면 알림은 스케줄링되지 않지만, 앱 자체는 정상 동작한다.

### 테스트 알림 (DEV 모드)

- 갱신 버튼을 길게 누르면 10초 후 테스트 알림이 발송된다.
- 알림 토글이 off이면 발송하지 않고 안내를 표시한다.

---

## 수정

- 제목, 시작일, 주기를 변경할 수 있다.
- 시작일은 오늘 이전 날짜만 선택할 수 있다.
- 시작일을 변경하면 경과일이 재계산되므로 진행도, 남은 일수, 만기 여부가 모두 바뀐다.
- 저장 시 알림이 재스케줄링된다.

> 예: 주기 30일인 알람의 시작일을 10일 전으로 변경 → 프로그레스바 33%, 남은 20일

---

## 삭제

- 알람 카드를 스와이프하여 삭제한다.
- 삭제 후 남은 알람 기준으로 알림이 재스케줄링된다.

---

## 알림 전역 토글

- 홈 화면 상단 스위치로 전체 알림을 on/off 할 수 있다.
- off: 모든 예약 알림이 취소된다.
- on: 알림 권한을 요청하고, 전체 알림을 재스케줄링한다.
